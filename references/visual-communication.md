# 复杂沟通可视化规则

## 触发条件

满足任一项且用图明显比纯文字更容易判断时，主动可视化，不等待用户点名：

- 3 张以上参考图，或参考图同时承担身份、风格、视觉系统等不同角色。
- 3 个以上构图/风格/文案方案需要横向比较。
- 前中后景、遮挡、叠压、透明度、裁切或图层顺序容易歧义。
- 主副标题、品牌、口号、microcopy、icon、logo、角标或注释的位置相互影响。
- 同一系统需要适配 9:16、16:9、1:1 等多种比例。
- 交付包含多个 prompt、多个版本、多个参考素材或多种输出格式。
- 用户已经反复用文字纠正同一空间、层级或关系问题。

一个简单问题、两个清楚的选项或一句话能解释的关系，不要可视化。

## 选择最小形式

| 问题 | 首选形式 | 必须显示 |
| --- | --- | --- |
| 参考图分工 | 参考角色矩阵 | 每张图的角色、学习项、禁用项、身份锚点 |
| 方案比较 | 并排比较或小矩阵 | 相同维度、关键差异、推荐及代价 |
| 图层前后 | 图层栈或爆炸图 | 从后到前顺序、遮挡关系、透明/穿插规则 |
| 文案与 icon 位置 | 标注线框 | 边界、锚点、对齐、留白、禁止覆盖区 |
| 多比例适配 | 同尺度构图草图 | 每个比例的主体、标题、留白和裁切变化 |
| 流程或依赖 | Mermaid 流程图 | 输入、拍板点、版本、下游交付 |
| 可调空间方案 | 交互式示意 | 只放当前真正需要调的控制项 |

静态关系能由标签和连线完整表达时用 Mermaid。需要显隐图层、切换比例、移动空间位置或比较状态时用交互式可视化。遵循 `$visualize` 的文件、主题、响应式和无障碍规则。

## 展示原则

- 一张图只回答一个主要决策问题；标题写在图外，图内只保留必要标签。
- 使用用户给出的真实参考编号、文案和图层名，不用“图 A/元素 1”等新别名制造二次理解成本。
- 不虚构审美评分、模型成功率或精确像素值。用户未定的内容标为“待定”，不要伪装成结论。
- 对空间示意使用相对区域和锚点，不把草图比例冒充最终像素坐标。
- 展示方案时保持比较维度一致；推荐必须同时显示其代价。
- 视觉展示是拍板界面，不是最终生图，也不替代 canonical JSON。

## 可视化后的收敛

1. 用一两句话说明图帮助用户决定什么，不复述全部标签。
2. 只问 1–2 个拍板问题，例如“选 B 的层级，还是保留 A 的留白？”
3. 把用户选择写回项目记录的“画面共识”以及纯 Prompt 的当前测试变量和 `prompt_template`。
4. 若选择改变核心瞬间、构图机制或风格家族，按大版本处理；局部位置和层级调整按小版本处理。

## 人类可读记录摘要

复杂交付把以下内容写进独立项目记录，按需删减；不要写进纯 Prompt：

```markdown
## 画面共识

- 核心瞬间：
- 参考图分工：
- 从后到前图层：
- 主体与关键物位置：
- 文案与 icon 位置：
- 多比例变化：
- 这版保留 / 舍弃：
```

摘要只记录最终决定，不逐字段翻译 JSON。对话中默认展示摘要和 Prompt 文件链接；完整 JSON 保存在独立纯 Prompt 文件中，需要核对时再展开。
