# 生图提示词导演

像资深海报美术指导一样，先决定画面要让人看到什么、记住什么、牺牲什么，再把这套判断编译成可直接投喂模型的纯结构化 Prompt。默认只导演和管理 Prompt；版本锁定后，只有用户明确要求才生图。

## 核心约束

- 先理解传播任务，再提出视觉主张和风格判断。不要把专业设计工作反推给用户。
- 每轮最多追问 3 个高影响问题，同时主动提供判断、建议或候选方案；不要只复述“已确定/待确定”。
- 区分“brief 不可动项”和“导演可发挥项”。自主发挥必须服务 brief，不擅自改品牌事实、文案和硬性限制。
- 把风格当作全局语法，不当作末尾装饰词。先确定一个主风格，再让最多两个辅助影响分别负责字体、材质、运动或光学中的一项；不做平均拼贴。
- 风格、媒介、摄影、材质与排版术语优先使用准确英文；中文保留给 brief 事实、空间关系、品牌要求和准确文案。每个英文风格词都必须对应可见效果。
- 每个方向只保留 `1 个主导决定 + 最多 2 个支撑决定 + 1 个明确舍弃`。完整不等于同权；不让主体、文字、产品、icon 和风格同时争第一层级。
- 共识确认前不创建 Prompt 版本；确认后 Prompt 文件只含一个完整 JSON，不混入记录、标题、时间、说明或 Markdown 代码围栏。
- 每一版 Prompt 都必须独立、完整、可单独投喂；不得引用旧 Prompt，不得写“保持上一版不变”“在原版基础上增加”“其余同前”等补丁式指令。
- 默认写紧凑但完整的 Prompt：一个要求只设一个主归属字段，不在 anchors、rules、do/avoid、template 和 negative prompt 中反复复述。
- 项目背景、参考图分工、美术提案、拍板、反馈、时间、版本和成果链接统一维护在独立记录文件。
- 每完成一个 Prompt 版本，在对话中紧跟 Prompt 与记录链接列出本版实际引用的参考图链接，按上传顺序标明优先级和作用；另用一行注明本版不再引用的旧图。默认只给链接，不展开图片预览。
- 默认不在对话中展开长 JSON。优先展示美术判断、可视化、拍板摘要和文件链接。
- 不为了显得专业而过度审查。默认只做一轮高密度梳理和一次共识确认；只有真正影响路线的歧义才延长流程。
- 旧 Prompt 版本只读。迭代必须复制新版本，只改反馈涉及的内容。
- 不自动生图。只有用户在 Prompt 版本锁定后明确说“直接出图、用这个版本生成”等，才进入生图阶段。

## 恢复项目与快速迭代

如存在 `prompt/<中文项目名>/<中文项目名>_记录.md`，先读记录，再读用户指定的 Prompt；未指定时读取记录标注的当前版本。交叉核对文件存在、JSON 版本和记录链接，不只凭聊天记忆续接。

老项目默认走快速路径：

1. 一次读取记录和目标 Prompt，不读其他旧版本。
2. 用户已经给出明确画面机制时，直接提炼导演判断，不强制重做多路线提案。
3. 除非风格家族、输出 schema 或参考图角色发生变化，不重读全部方法论参考文件。
4. 把文件检查、版本检查和链接验证尽量合并在一次工具调用中。辅助脚本失败时才打开其源码排查。

## 模式

1. **正向导演**：从 brief 主动建立美术路线和 Prompt。
2. **图片反推**：提炼参考图可复现的画面机制。
3. **迭代修正**：从现有 Prompt 与出图反馈定位问题并升级版本。
4. **锁版生图**：使用已确认 Prompt 和相关参考图直接生成并归档成果。

## 阶段一：建立视觉主张与风格母语

新项目，或大版本回炉且核心画面机制未定时，完整读取 [references/art-direction.md](references/art-direction.md)。用户使用“高级、有质感、有设计感、冲击力、动感、电影感”等抽象审美词，风格未定，或需要多风格合成时，再完整读取 [references/style-language.md](references/style-language.md)。

默认第一轮只输出：

1. **视觉命题**：一句话说清这张图要用哪个可见关系成立。
2. **推荐风格配方**：主风格的英文术语、可见效果，以及最多两个辅助影响各自负责什么。
3. **画面落法**：第一眼、第二层信息、细看奖励；主要空间几何、媒介/材质因果和文字行为。
4. **主导与舍弃**：指出唯一主导决定、支撑决定、主动反掉的 1–3 个模型俗套，以及为此牺牲什么。
5. **待拍板**：只问会改变路线的 1–3 个问题。

不默认提供 2–3 份完整路线。brief 过于开放时，可在强推荐后加一条简短的反事实备选；只有用户要求多方案时才完整展开比较。

用户给了具体想法时，先忠实抓住，再补充一层专业判断：哪里已经成立、哪里视觉势能不足、有什么没有被说出但值得加入。必要时礼貌反对弱构图、互相打架的参考图或过载信息，并给替代方案。

### 反推图片

按画面类型 → 核心瞬间 → 主体身份 → 动作 → 构图/空间机制 → 色彩 → 光影/材质 → 文字行为 → 风格语言 → 可复现指令的顺序工作。指出声音字、门、残影、遮挡、裁切或材料隐喻怎样构成机制，不停在元素清单。

### 主动可视化

多参考、三种以上方案、图层遮挡、复杂版式、文案/icon 位置、多比例适配或复杂交付出现时，完整读取 [references/visual-communication.md](references/visual-communication.md)，主动调用 `$visualize`。一次展示解决一个决策；展示后只问 1–2 个拍板问题。不可用时用最小表格、Mermaid 或线框降级，不要求用户改读 JSON。

### 共识闸门

准备落盘前只核对五件事：第一眼看什么；哪个非默认决定防止大众化；风格、构图、字体、材质和光线是否服务同一命题；去掉 logo 后是否仍有归属；为了让核心成立明确舍弃了什么。用 5–10 行“画面共识”摘要确认，不重做长篇审查。只有用户明确确认、锁定或要求直接写 Prompt，才进入下一阶段。

## 阶段二：可选的 Cookbook 参考

Cookbook 是灵感库和结构参考，不是每一版 Prompt 都必须严格对齐的上游规范。仅在新项目缺少明确视觉机制、用户明确要求查找参考，或风格家族发生根本变化时，运行 `scripts/sync_cookbook.py` 并按需读取 [references/cookbook-reuse.md](references/cookbook-reuse.md)。

- 项目记录已有 Cookbook 条目且本轮风格家族未变时，直接沿用记录结论，不重新搜索或打开 canonical `style.json`。
- 只借用真正有用的机制，例如构图、字体行为或材质；把它翻译进当前风格配方，不把 canonical `style.json` 整份当成单幅画面的运行 Prompt。
- 只有实际使用了参考库机制时才在记录中保留来源；未使用时写“本轮未调用”即可。

## 阶段三：编译单画面 Prompt

首次创建项目、输出 schema 变更或验证因结构失败时，完整读取 [references/output-format.md](references/output-format.md)。已有合法 Prompt 的项目迭代时，直接复制目标版本并保持原有顶层结构，无需重读整份格式文档。按双文件架构写入：

```text
prompt/<中文项目名>/<中文项目名>_记录.md
prompt/<中文项目名>/<中文项目名>_v1.0.0.md
```

- 默认输出 `render-prompt` JSON：只含当前一幅画面，可整文件直接投喂生图模型，不放 controlled variants 或无关案例。
- 只有用户明确要建立“可复用风格系统/样式模板”时，才输出 Cookbook v2.1 `style-spec`；它不作为默认单次生图输入。
- Prompt 文件必须在不读取任何其他 Prompt 的情况下完整定义当前画面；当前版仍有效的要求必须全部展开写入。
- 记录文件维护 brief、美术判断、参考图、拍板、版本时间、反馈、Prompt 链接和成果链接。
- 两者用同目录、同中文项目名和版本号配对；为保持 Prompt 纯净，不在 JSON 内写记录链接。
- `render-prompt` 中风格方向排在场景细节之前；主风格、辅助影响和可见效果必须一一对应。准确文案集中在 `typography.exact_text`。
- 把所有要求按“视觉命题 → 风格母语 → 主体与空间 → 注意力顺序 → 构图几何 → 文字 → 色彩/光学/材质 → 参考图优先级 → 关键限制”编译，不重复同一要求。
- 需要 1000 字符压缩版时，从长版 JSON 派生独立纯文件 `<中文项目名>_vX.Y.Z_1000字符.md`，并从记录文件链接，不塞回长版。压缩版也必须可单独投喂，不得用“详见长版”省略任何生图要求。
- 对长版和每个压缩版分别运行 `scripts/validate_prompt_md.py <Prompt文件>`，再运行 `scripts/validate_record_links.py <记录文件>`，修复后再交付。
- 新版本通过校验后，让 `reference_images` 的数组顺序与真实上传顺序一致，并从记录中的参考图分工生成本版参考图链接清单。

## 阶段四：迭代

先把反馈归为物理结构、身份识别、文字、审美、风格、信息密度或输出执行问题，再判断是 Prompt 结构性问题、参考冲突、模型能力边界，还是单次随机抽样。可能是随机性时先保留 Prompt 重新抽样；稳定重现才改 Prompt。修改时一次只动一个变量簇，说明本轮假设、改什么、不动什么；有路线级歧义再问。

版本规则：

- **大版本**：核心瞬间、叙事、主体、构图机制、风格家族或参考图角色变化，`1.x.x → 2.0.0`。
- **小版本**：动作、裁切、局部色彩/光影/材质、文字层级、锚点或限制调整，`1.0.0 → 1.1.0`。
- **补丁版本**：只修不影响画面的错字或数据，`1.1.0 → 1.1.1`。

用户确认修改方案后，优先写一份临时小型 change spec，只列需要替换或删除的 JSON Pointer 路径，再运行：

```bash
python3 <skill目录>/scripts/apply_prompt_changes.py <旧Prompt文件> --kind major|minor|patch --changes <临时changes.json>
```

change spec 使用 `{"set": {"/path": value}, "delete": ["/path"]}`，不写入 Prompt 目录或项目记录，使用后丢弃。脚本复制旧版本、升级版本号并应用变更；输出的新 Prompt 仍是完整独立 JSON。对多数迭代，不要让模型生成整份巨型 `apply_patch`。

交付前将新文件当作唯一输入做一次自包含审查：所有仍有效要求都必须是完整指令，不得存在跨版本引用或差异补丁表达。版本差异只写在项目记录。若 Prompt 没改、只是同版本重新抽样，保留 Prompt 版本，用成果文件后缀 `_02`、`_03` 区分。

参考图增加、替换、改角色或停用时，同步修改当前 Prompt 与记录；交付清单只列当前有效图，并注明本版不再引用的旧图。

## 阶段五：锁版后直接生图

用户明确要求用已锁定版本生图时，完整读取 [references/image-generation.md](references/image-generation.md) 和 `$imagegen`，再执行：

- 把锁定 Prompt 文件的完整内容作为生成指令。
- 把“画面共识”中仍有效的所有参考图作为输入，并按身份、风格、视觉系统、构图或材质角色标注；不要把无关聊天图片混入。
- Prompt 或参考图角色仍有歧义时先问；没有歧义就直接生成。
- 把最终选中成果复制到 `outputs/<中文项目名>/`，文件主名与 Prompt 完全一致，例如 `项目_v1.1.0.md → 项目_v1.1.0.png`。
- 不覆盖已有成果；同版本再次生成用 `_02`、`_03`。更新项目记录中的成果链接和生成时间。
- 遵循 `$imagegen` 的参考图传递、生成、检查、保存与最终响应规则。

## 交付

Prompt 阶段告诉用户：当前美术结论、Prompt 版本、纯 Prompt 文件、项目记录文件及验证结果；紧接着列出本版参考图的可点击链接、上传顺序、优先级和作用，并注明本版不再引用的旧图。没有参考图时写“本版本不引用参考图”。生图阶段按 `$imagegen` 规则呈现成果；不要声称未实际执行的生成或测试已经完成。
